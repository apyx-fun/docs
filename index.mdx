## What is Apyx?

Apyx is a **permissionless token launch layer** on Solana. Tokens launch on bonding curves (Pump-style). When two tokens land in the same market-cap band and time window, they **duel**: first to reach a target market cap triggers resolution; the **higher market cap wins**. The loser’s SOL (minus a protocol fee) moves to the winner’s curve and is converted into winner tokens for a **mercy vault**. Loser-token holders can **claim mercy**—send loser tokens in, receive winner tokens pro-rata. So value is reallocated to the winner instead of being wiped.

## The problem

- **Fragmented launches.** Many tokens, similar size, no way to surface which ones have real demand.
- **No skin in the game.** Launch platforms don’t tie outcomes to measurable performance.
- **Opaque or controllable “contests.”** If pairing or winners can be chosen off-chain, the game isn’t fair.

---

## The solution

1. **Bonding-curve launches** — Same primitive as Pump: buy/sell on a curve; market cap is derived from reserves.
2. **Deterministic duels** — Pairing is **on-chain and rule-based**: same market-cap bucket + same time window + **match key = Blake3(mint, bucket, window)**. Collision = duel. No operator chooses who faces whom.
3. **Clear win condition** — Target = 110% of the bucket’s reference mcap. When any trade pushes either token’s mcap past that, the program resolves: **higher mcap wins**. Relayer only submits the tx; the program enforces the rule.
4. **Mercy** — Losers don’t get zero. They get a **pro-rata claim on winner tokens** at the on-chain rate (loser SOL → winner tokens in the mercy vault). You’re “joining the winner” at a fixed, verifiable conversion.

---

## In one sentence

**Apyx turns token launches into verifiable, permissionless duels: pairing and winners are determined by on-chain rules (bucket + window + hash collision; higher mcap wins), and loser holders can claim winner exposure via mercy.**
